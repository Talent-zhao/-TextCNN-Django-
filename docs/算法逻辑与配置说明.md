# 算法逻辑与配置说明

本文档描述**当前仓库实现**中的文本分类、多模型融合、规则打分、训练与评估落库等算法步骤及配置项含义。  
论文向的概览仍可参考 `docs/3.3-算法应用.md`；**以后台实际代码为准时以本文为准**。

## 1. 总体数据流

1. **输入**：原始字符串（前台 `/predict/`、后台 `RawText.content` 等）。
2. **预处理**：`algorithm/text_utils.preprocess_pipeline`（正则清洗、jieba 分词、停用词表 `stopwords.txt` 等，与训练侧一致）。
3. **子模型推理**：`PredictService.build_result_map` 并行调用 TF-IDF+sklearn（多角色）、TextCNN、TextRCNN、关键词规则，得到统一结构的 `dict`（含 `enabled`、`available`、`prob` 或 `score`、`label` 等）。
4. **融合**：`PredictService.fuse_prediction_result` → `algorithm.fusion.fuse_weighted_scores`，得到 **最终风险分 `final_risk_score`** 与 **离散 `risk_level`（0～3）**。
5. **展示标签**：在各子模型预测概率中取最大者对应类别名 `_best_label_from_models`，再经 `LabelDefinition` 解析为库内标签（`_resolve_label`）。
6. **持久化（仅后台流水线）**：`PredictionResult` 写入融合分、规则分、是否过线标记 `is_alert_triggered`、`detail_json` 等；`RawText` 状态流转为「已预测」。

**核心代码**：`app/services/predict_service.py`、`algorithm/fusion.py`、`algorithm/rules.py`、`algorithm/sklearn_tfidf.py`、`algorithm/torch_nlp.py`。

## 2. 文本预处理（与训练对齐）

- **函数**：`algorithm/text_utils.preprocess_pipeline(text)`。
- **用途**：sklearn 推理前对单条文本做与训练脚本相同的清洗与分词管线，保证 TF-IDF 输入分布一致。
- **停用词**：项目根目录 `stopwords.txt`。

## 3. 前台在线预测（不写业务预测表）

| 步骤 | 说明 |
|------|------|
| 1 | 视图 `app/views.py` 中 `predict.get` 列出 `ModelInfo` 中 `status='ready'` 的模型供下拉选择；默认选中当前「启用」模型。 |
| 2 | `predict.post` 读取 `content` 与可选 `model_id`。若 `model_id` 对应一条 `ready` 模型，则 **`PredictService.predict_text_line(content, model_info=该条)`**；否则 **`model_info=None`**，内部仍走 `load_active_model()`（与后台流水线默认一致）。 |
| 3 | `load_active_model()`：取 `is_active=True` 且 `status=ready` 的最新一条；若无则创建/启用默认 `svm_baseline`。 |
| 4 | `PredictService.get_runtime_config(model_info)`：合并 `SystemConfig`、按 `model_type` 覆盖子模型开关、再合并 `model_info.config_json`。 |
| 5 | `predict_text_line`：`build_result_map` → `fuse_prediction_result` → `_best_label_from_models`，返回 `label`、`final_score`、`risk_level`、`submodels` 及所用模型名称/版本/类型等。 |
| 6 | `JsonResponse` 返回 `msg`、`final_score`、`risk_level`、`model_*` 等；Ajax 携带 CSRF。 |

## 4. 后台 RawText 批量/单条预测

| 步骤 | 说明 |
|------|------|
| 入口 | `PredictService.run_prediction_for_rawtext(rawtext_id)`（`app/riskViews.py` 等待预测列表触发）。 |
| 状态 | 要求当前为 `pending_predict`；若仍为 `labeled` 会先自动转为 `pending_predict`。 |
| 推理 | 与 §3 相同：`build_result_map` + `fuse_prediction_result` + `_best_label_from_models` + `_resolve_label`。 |
| `probability_score` | 各 **可用** 子模型 `prob` 的 **最大值**（`_max_model_prob`），用于结果表展示，**不等于**融合分。 |
| `rule_score` / 命中词 | 来自 `predict_rules` 的 `score` 与 `hit_words`。 |
| 写库 | `save_prediction_result` 创建 `PredictionResult`；再组装 `detail_json`（含 `submodels`、`fusion` 权重与阈值快照）；`RawText.transit_to('predicted')`。 |
| 触发说明文案 | 若 `is_alert_triggered` 为真，`_compute_alert_trigger_reason` 根据 `risk_level`、规则分与命中词生成简短原因写入 `detail_json['alert']`。 |

## 5. 子模型说明

### 5.1 TF-IDF + sklearn（五种角色）

- **角色**：`svm`、`knn`、`rf`、`dt`、`lr`，与 `ModelInfo.model_type` 及磁盘文件名 `model/model_*.pkl` 对应。
- **推理**：`algorithm/sklearn_tfidf.predict_tfidf_sklearn`：加载 `joblib` 分类器与向量器，`transform` 后 `predict` / `predict_proba`；`prob` 取 **最大类概率**。
- **向量器路径解析**（`PredictService._resolve_vec_path`）：  
  - 若当前启用模型恰好为该角色且登记了 `vectorizer_path`，优先用之；  
  - 否则尝试 `model/tfidfVectorizer_{role}.pkl` 存在则用之；  
  - 否则回退 `tfidf_vectorizer_path` / 模型登记 / 默认 `model/tfidfVectorizer.pkl`。  
  **目的**：多算法各用各次训练产出的词表，避免「分类器与向量器不匹配」。

### 5.2 TextCNN / TextRCNN

- **入口**：`algorithm/torch_nlp.predict_textcnn` / `predict_textrcnn`。
- **配置**：权重路径、词表路径、`num2name.json`、以及 `textcnn_*` / `textrcnn_*` 结构超参（见 §8）。
- **返回**：与其它子模型一致的 `enabled` / `available` / `label` / `prob` 结构。

### 5.3 关键词规则（不参与 sklearn 训练）

- **实现**：`algorithm/rules.predict_rules`。
- **词表**：高风险词 `HIGH_RISK_RULE_WORDS`、负面词 `NEGATIVE_WORDS`（见 `algorithm/rules.py`）。
- **打分**：命中高风险词每个 **+0.25**，负面词每个 **+0.08**，总分 **上限截断为 1.0**；返回 `score` 与 `hit_words`。
- **融合时**：作为一路子模型，使用配置键 `weight_rules`，分数字段为 `score`（与 `prob` 并列由融合层读取）。

## 6. 融合算法与风险等级

### 6.1 加权融合

- **函数**：`algorithm/fusion.fuse_weighted_scores(parts, cfg)`。
- **输入**：`parts` 为列表 `[(score, weight), ...]`，仅包含已启用、可用、且权重 &gt; 0 的子模型；`score` 为 0～1 的 `prob` 或规则 `score`。
- **公式**：  
  `final_score = Σ(score_i × weight_i) / Σ(weight_i)`（权重和为 0 时按实现退化为避免除零）。
- **组装列表**：`PredictService.fuse_prediction_result` 按固定顺序遍历 `svm, knn, rf, dt, lr, textcnn, textrcnn, rules`，读取 `weight_*` 配置。

### 6.2 风险等级 `risk_level`（0～3）

在 `fuse_weighted_scores` 内根据 **`final_score`** 分档（注意：**中间两档阈值为代码写死**，与 `threshold_alert` 无关）：

| 条件 | `risk_level` |
|------|----------------|
| `final_score >= threshold_high_risk`（默认 0.85，来自 `cfg`） | 3 |
| `final_score >= 0.65` | 2 |
| `final_score >= 0.4` | 1 |
| 否则 | 0 |

若需让「中高风险」完全由配置驱动，需同时改 `algorithm/fusion.py` 与本文档 §6.2。

## 7. 「是否触发预警」字段（`PredictionResult.is_alert_triggered`）

此为 **业务标记字段**（分数/等级过线），**不是**已删除的「预警中心」模块。

在 `PredictService.save_prediction_result` 中，满足 **任一** 即为真：

- `final_risk_score >= threshold_alert`（默认 **0.65**，配置键 `threshold_alert`）；或
- `risk_level >= 2`；或
- `rule_score >= threshold_rule_alert`（默认 **0.5**，配置键 `threshold_rule_alert`）。

**触发原因文案**（`detail_json`）：`_compute_alert_trigger_reason` 按「模型高风险 / 规则为主 / 组合」与命中词拼接，供预测详情页展示。

## 8. 运行配置项（`SystemConfig` / 预测融合设置页）

配置由 `PredictService.ensure_default_runtime_config` 首次写入数据库，后台「预测融合设置」可改。默认值见 `PredictService.DEFAULT_RUNTIME_CONFIG`（`app/services/predict_service.py`）。

### 8.1 子模型开关

| 键 | 含义 |
|----|------|
| `enable_svm` … `enable_lr` | 是否参与该路 TF-IDF+sklearn 推理与融合。 |
| `enable_textcnn` / `enable_textrcnn` | 是否参与深度学习两路与融合。 |
| `enable_rules` | 是否启用规则分。 |

### 8.2 融合权重（与开关同时生效）

`weight_svm`、`weight_knn`、`weight_rf`、`weight_dt`、`weight_lr`、`weight_textcnn`、`weight_textrcnn`、`weight_rules`。  
权重 ≤ 0 的路由在融合时被跳过。

### 8.3 阈值

| 键 | 默认 | 用途 |
|----|------|------|
| `threshold_high_risk` | 0.85 | 融合分达到则 `risk_level=3`（§6.2）。 |
| `threshold_alert` | 0.65 | `is_alert_triggered` 条件之一（§7）。 |
| `threshold_rule_alert` | 0.5 | 规则分达到则 `is_alert_triggered`（§7）。 |

### 8.4 模型与词表路径

`tfidf_vectorizer_path`、`svm_model_path` … `lr_model_path`、`textcnn_vocab_path`、`textcnn_weight_path`、`textrcnn_vocab_path`、`textrcnn_weight_path`、`num2name_path` 等；以及 TextCNN/TextRCNN 的结构超参（`textcnn_max_len`、`textcnn_num_classes` 等）。

### 8.5 按「当前启用模型类型」强制子模型开关

`get_runtime_config(model_info)` 在存在 `model_info` 时，会按 `model_type` **重写**各 `enable_*`：

- 单类型（如 `svm`、`textcnn`）：仅打开对应 sklearn 角色或单一路深度学习 + 通常保留规则。
- `fusion`：五 sklearn + textcnn + textrcnn + rules 全开（具体仍以合并后的 `config_json` 为准）。

**新增 `model_type` 时**：除实现代码外，须更新本文 §8.5 与 `PredictService.get_runtime_config` 分支说明。

## 9. 训练流程与评估落库

### 9.0 后台「模型训练」页的 CSV 列表（`app/trainViews.py`）

- 数据按语言分目录：`datasets/depression_nlp/en/`（英文）、`datasets/depression_nlp/zh/`（中文）；默认训练 CSV 为 `zh/oesd_keyword_binary/splits/train.csv`（见 `trainViews.DEFAULT_TRAIN_CSV`）。
- 自动扫描 `datasets/depression_nlp/**/*.csv` 填充下拉框时，**不会列出**验证/测试划分文件，避免误用：  
  - 文件名为 `val.csv`、`test.csv`、`validation.csv`、`dev.csv`（不区分大小写）；或  
  - 相对路径中**任一级目录名**为 `val`、`test`、`validation`、`dev`。  
- 典型保留项：各数据集下的 `splits/train.csv`、`raw/*.csv` 等。若修改过滤规则，须同步更新本文与 `training_hub.html` 提示文案。

### 9.1 单算法 sklearn：`scripts/train_single_sklearn.py`

1. 读 CSV（列 `text`、`label`），预处理与 `LabelEncoder`。
2. `train_test_split` 划分训练/验证（默认 80/20，尽量分层）。
3. 在训练子集上 `TfidfVectorizer.fit_transform`，验证子集 `transform`；训练分类器；**计时**得到 TF-IDF+拟合墙钟时间 `training_time_sec`（写入评估）。
4. 验证集：`predict`、`predict_proba`；`classification_report` 得 macro P/R/F1；**AUC**：`app.training_evaluation.roc_auc_from_proba`（二分类用正类概率；多分类 OvR macro）。
5. **过拟合**：`infer_overfitting_from_train_val_acc`——训练子集准确率与验证集准确率之差 **&gt; 6%** 为「是」，否则「否」。
6. 全量数据重新 `fit` 向量器与分类器并保存 `model/` 下标准文件名；注册 `model_training_registry.json`。
7. `save_evaluation_from_training` 写入 `ModelEvaluation`（含 `auc`、`training_time_sec`、`is_overfitting`）。

### 9.2 TextCNN / TextRCNN：`scripts/train_char_torch.py`

1. 字符级词表与 `CharDataset`，划分 train/val。
2. 多 epoch：训练循环 + 验证集 `argmax` 准确率与分类报告；最后一轮验证收集 **softmax 概率矩阵** 算 AUC；最后一轮在训练集上再跑一遍算训练准确率，与验证准确率比较差值得过拟合判定。
3. 全量再训一轮后保存权重与词表；`save_evaluation_from_training`；**训练耗时**为各 epoch 训练循环墙钟时间之和（不含最后全量写盘段，见代码注释习惯）。

### 9.3 多模型批训：`scripts/train_sklearn_classifiers.py`

一次训练 KNN/RF/DT/LR 等并写盘；**不**划分验证集、**不**调用 `save_evaluation_from_training`。若需后台「评估结果/算法对比」出现指标，请对单算法使用 `train_single_sklearn.py`。

### 9.4 环境变量

- `DEPRESSION_WEB_SKIP_TRAIN_EVAL`：设为 `1`/`true`/`yes` 时跳过写 `ModelEvaluation`。

### 9.5 清理不完整旧评估记录

升级训练脚本前写入的 `ModelEvaluation` 可能缺少 `auc`、`training_time_sec`。可执行：

```bash
python manage.py cleanup_incomplete_train_evaluations --dry-run   # 预览
python manage.py cleanup_incomplete_train_evaluations              # 删除
```

仅删除 `model_version` 以 `train_` 开头且 **AUC 与训练耗时均为空** 的记录，不影响已补全指标的新记录。

## 10. 评估结果与算法对比页

- **模型**：`ModelEvaluation`（准确率、精确率、召回率、F1、可选 AUC、训练秒数、`is_overfitting` 等），外键 `ModelInfo`。
- **源算法对比表**（`AlgorithmCompareView` → `build_algorithm_compare_rows`）：对固定七种 `model_type`（`svm, knn, dt, rf, lr, textcnn, textrcnn`）各取 **该类型下最新一条** `ModelEvaluation`；缺项再尝试对应 `ModelInfo.metrics_json` 中的同名字段（如 `auc`、`roc_auc`、`training_time_sec`、`is_overfitting` 等）。评估记录由训练脚本写入；补充指标可编辑「模型管理」中的 `metrics_json`。
- **融合策略对比宽表**（`build_fusion_strategy_compare_rows` → `app/services/fusion_strategy_report.py`）：每条 `model_type=fusion` 一行（排除占位 `线下评估记录` / `default`）。**运行时 cfg** 由 `PredictService.get_runtime_config(model_info)` 得到：**先**合并该条 `metrics_json.fusion_snapshot`（登记快照），**再**合并 `config_json`（覆盖快照）。全局「预测融合设置」同步仅更新 **名称=「融合策略」且 version=default** 的 `ModelInfo.metrics_json`；其他名称的融合登记不会被 `sync_modelinfo_paths_from_fusion_config` 整批覆盖，保存模型后会按当前有效 cfg 单独重写其 `fusion_snapshot`。宽表指标、留一与缓存（`fusion_strategy_report:v4`）同前文。
